Executive Summary

일감 트래커가 아니다 — 게임 QA 사이클 전체를 담은 도메인 특화 업무 플랫폼.

QA Workflow Management는 게임 QA 조직의 실무 요구사항을 기반으로 설계된 업무 관리 플랫폼이다. 단순한 일감 트래커를 넘어, 게임 QA의 특수성 — 빌드 사이클 · 테스트 유형 분리 · 팀 상태 가시화 · 근태 연동 · 리스크 자동 계산 — 을 반영한 도메인 특화 솔루션이다.

하나의 일감을 생성하고 완료하는 것으로 끝나지 않는다. 빌드 수급 → 테스트 수행 → 이슈 발견 → 배포 → 라이브 회고의 QA 사이클 전 구간을 단일 인터페이스 안에서 이어 붙여, 데이터가 사이클 사이에서 끊기지 않도록 설계했다.

JIRA + 엑셀 + 슬랙으로 분산되어 있던 운영을 통합하고, 기한과 진척도를 기반으로 리스크 레벨을 자동 계산하며, 팀 상태 · 근태 · 회의실 · 알림까지 하나의 실시간 소켓 위에서 동기화한다. 결과적으로 "어떤 빌드의, 어떤 일감이, 지금 누구에 의해, 어떤 리스크로 진행되는가" 를 단일 화면에서 즉시 파악 가능하다.

개요 Domain · Problem · Philosophy

Why게임 QA 도메인

QA Workflow는 게임 QA 조직의 실무 요구사항을 기반으로 한 업무 관리 플랫폼이다. JIRA · 엑셀 · 슬랙으로 분산되어 있던 빌드/일감/회고 데이터를 단일 실시간 인터페이스로 묶는다.

게임 QA의 특수성이 제품 전반을 관통한다 — 빌드 차수별 수급, 9종 테스트 유형 개별 진척, 라이브 이슈와 업데이트 타겟의 매핑, 팀원 상태의 실시간 가시화, 근태와 타이머 연동.

How해결하는 문제

  • 분산 운영: JIRA + 엑셀 + 슬랙 → 단일 통합 인터페이스
  • 수동 리스크 → 기한/진척도/빌드 상태 기반 자동 계산
  • 정적 팀 상태 → 픽셀 오피스 + 실시간 소켓
  • 텍스트 공지 회의 → 픽셀 오피스 내 채팅 회의실
  • 별도 근태 시스템 → 일감 타이머와 자동 연동
  • 배포 후 회고 산발 → 타겟별 기록 카드 + 트렌드

01빌드 사이클 = 회고 단위

UPDATE 타겟 1건 = 기록 카드 1장. 같은 기간 내 다중 UPDATE 는 각각 독립 카드로 회고. HOTFIX는 UPDATE의 서브셋으로 현재 윈도우에 귀속되어 집계된다.

02stats 위치 고정

이슈/커뮤니티 stats는 활성 UPDATE 빌드 ID에 누적 저장. HOTFIX가 배포되어도 회고 데이터가 "사라진 듯" 보이지 않도록 보존된다.

03실시간 동기화

REST로 읽고 쓰되, Socket.io로 전 세션 동시 갱신. 일감 상태 · 팀 상태 · 회의실 입퇴장 · 빌드 전이가 모두 다른 화면에 즉시 반영된다.

기술 스택 Full-Stack Overview

Frontend

React 19 TypeScript 5.9 Vite 8 Tailwind v4 Zustand 5 TanStack Query 5 React Router 7 Framer Motion 12 Recharts 3 Socket.io Client 4.8 React Hook Form + Zod

TanStack Query로 서버 state 캐싱 + invalidation을 위임하고, Zustand는 auth/UI 전역 state만 관리. Redux 보일러플레이트를 제거.

Backend

Node.js 24 TypeScript ESM Express 5 Prisma 6.19 Socket.io 4.8 JWT bcryptjs Zod 4 Helmet · CORS · Rate Limit

Express 라우트 + Socket.io를 동일 프로세스에서 운영. Prisma는 Type-safe + 마이그레이션 관리 + Supabase 호환.

데이터 & 외부 연동

PostgreSQL (Supabase) JIRA MCP Confluence Wiki Epic7 PHP Admin CZN Jenkins TestRail API

모든 외부 연동은 환경 변수로 토큰 격리. MCP Streamable HTTP로 사내 JIRA/Wiki 접근, Jenkins/PHP는 서버 프록시.

인프라 & 운영

pm2 (ecosystem.config) Vite dev server (LAN) Optimistic Lock FSM 검증 Zod 전수 검증 ActivityLog 감사

pm2 상시 구동 + Vite HMR로 팀원이 LAN IP 직접 접근. 수정은 재시작 없이 Ctrl+F5로 반영되는 게 기본.

Design SystemTheme 3종 · Meaningful Motion

Medical Monitor (ECG 파형 + cockpit 헤더) · Game RPG (Stardew / Minecraft / Notebook 팔레트 3) · Silicon Valley Cyberpunk (AI 로봇 사이드바 + 코드 패턴 벽지). 모든 애니메이션은 실 데이터에 매핑된 의미 있는 모션이다. 라이트/다크 자동 대응.

5.1  컨테이너 다이어그램

Container Diagram · 3 tiers · 5 external systems · pm2 orchestration

Notepm2 + Vite HMR

  • 운영 PC에서 pm2 start ecosystem.config.cjs 상시 구동
  • 2개 앱: qa-workflow-server (port 4000) + qa-workflow-client (port 5176)
  • 팀원이 LAN IP로 접속, 코드 변경은 HMR로 재시작 없이 반영

NoteREST + WebSocket 동일 포트

  • Express 5 라우트는 /api/*로 REST 제공
  • Socket.io는 동일 HTTP 서버에 attach, JWT 미들웨어로 검증
  • Room 기반 브로드캐스트: global · meeting:N

데이터 파이프라인 Request · Validation · Broadcast

사용자 액션 → 검증 → 서비스 → DB → 실시간 브로드캐스트 순으로 단방향 흐름을 유지하고, 그 결과가 다른 모든 세션에 자동 반영된다. 모든 상태 변경은 FSM + Optimistic Lock + ActivityLog 3중 게이트를 통과한다.

상태 변경 예시 · 일감 상태 전이 + 실시간 브로드캐스트

Sequence · Task Status Transition · 6 steps with Socket broadcast

Gate 1FSM 검증

VALID_TRANSITIONS에 허용된 전이만 통과. Task 9가지 상태, Build 6단계, 회의실 상태까지 서버에서 엄격히 검증.

Gate 2Optimistic Lock

version 필드로 동시 편집 conflict 감지. 사용자 A · B가 같은 일감을 동시 수정하면 후착 트랜잭션 자동 거부.

Gate 3ActivityLog 감사

모든 상태 변경은 ActivityLog에 기록. 누가 · 언제 · 무엇을 · 왜 변경했는지 추적 가능. 회고 / 감사 / 보안 점검용.

자동 리스크 계산 (risk.service.ts) 

기대 진척도(expected) = (today - startDate) / (dueDate - startDate) × 100
실제 진척도(total)    = progressTotal
차이                 = expected - total

차이 > 40% && 기한 임박 → CRITICAL
차이 > 25%              → HIGH
차이 > 10%              → MEDIUM
else                    → LOW
